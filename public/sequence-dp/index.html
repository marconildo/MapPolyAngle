Optimizing Flight Area Traversal with Dynamic Programming

This article explains the sequencing layer in the mission planner as it exists now: once a large mapping site has been partitioned into survey areas and each area already has an optimized local bearing, the app still has to decide the order in which those areas should be flown and whether each area should be reversed. The same backend optimizer is used by the in-memory Optimize Transit workflow and by export. It supports both area-only missions and imported Wingtra missions with valid takeoff and landing anchors.

1. Problem setup

Mapping in changing terrain

The operational objective is simple to state but hard to execute well: cover as large an area as possible with consistent GSD and acceptable mission time. A single large survey polygon, like luz0.flightplan, is a natural starting point. It can target a nominal camera ground resolution of 1.39 cm/px, but one large survey block forces one global traversal pattern onto terrain that is not globally uniform.

In production we therefore do something else first: we partition the original site into smaller areas and optimize a local survey bearing inside each one. A real example is luz0-realProd.flightplan, which replaces the single original block with 17 sub-areas.

That leaves a different optimization problem. The earlier algorithms make the mapping itself better: more consistent GSD, more suitable local bearings, and more manageable survey blocks. The sequencing layer then makes the resulting mission cheaper to fly by reducing long transfers, avoidable climbs, and bad area-to-area ordering.

In other words, the sequencing layer receives a compressed mission rather than the full geometric planning problem. Each area has already been reduced to two traversal modes: fly the fixed area path forward, or fly that same path flipped. The optimizer then chooses a global order and a flip state for every area. If the imported flightplan has valid takeoff and landing items, those endpoints are included as optional depot constraints on the first and last legs. If it does not, the same solver naturally falls back to a free-start, free-end area sequence.

  1. Start from a large region that needs to be mapped accurately.
  2. Partition it into smaller terrain-aware survey areas.
  3. Optimize the local flight bearing inside each area.
  4. Only then solve the global traversal order and flip decisions.

2. What dynamic programming is

Dynamic programming reuses repeated subproblems instead of rediscovering them

Dynamic programming is a method for solving a large optimization problem by expressing it as a family of smaller problems, solving each distinct smaller problem once, and reusing the result. The central signal that DP is appropriate is that brute force keeps revisiting the same sub-question.

Overlapping subproblems

Different partial routes often collapse to the same “visited set + current terminal area + flip state”.

Optimal substructure

If the best full mission ends in a given state, the prefix reaching that state must also be optimal.

A useful way to phrase it is: state is the smallest amount of information that fully describes the subproblem. Once the state is chosen well, the recurrence usually becomes obvious.

3. Why DP is relevant here

This sequencing problem has exactly the structure DP likes

The global problem is TSP-like, but not identical to the textbook tour formulation. We are solving for a Hamiltonian path through the survey areas, not for a closed area-to-area cycle. In the simplest area-only case that path is free to start and end at any area. In an imported Wingtra mission, valid takeoff and landing endpoints become optional depot constraints: a takeoff endpoint constrains the first leg, a landing endpoint constrains the last leg, and either one can be present without the other.

For 11 areas, a brute-force solver would examine every permutation of areas and every independent flip assignment. That is 81,749,606,400 mission variants before considering any depot anchor. The anchors change the score of each candidate path, but they do not change the fundamental combinatorics: each area still contributes one position in the permutation and one binary flip choice.

Brute-force search space
Candidates(N) = N! · 2N
  • The order history matters only through the set of areas already visited.
  • The continuation cost depends on the area we are currently ending on.
  • The continuation cost also depends on whether that terminal area was flown forward or flipped.
  • Optional takeoff and landing endpoints affect only the base and final edge costs.
  • The continuation does not depend on the exact route prefix beyond those facts.

4. Model and recurrence

Once the areas are fixed, the sequence optimizer becomes a subset DP

Each area contributes exactly two traversal modes: forward and flipped. Both modes share the same already-optimized internal strip geometry; flip only changes which end of that fixed pattern is used as ingress and which end is used as egress.

State
DP[S, i, f]

Minimum transfer cost after visiting exactly the set S and ending at area i with flip state f.

Base case
DP[{i}, i, f] = B(i,f)

B(i,f) is the optional takeoff-to-area cost. For area-only missions it is zero, so free-start sequencing is just a special case.

Transition
DP[S ∪ {j}, j, g] = mini ∈ S, f ∈ {0,1} DP[S, i, f] + C[(i,f) → (j,g)]

Every transition uses a precomputed connector cost between the end of one traversal mode and the start of another.

After all areas have been visited, the solver adds an optional final edge from the last area mode to the landing endpoint. With no landing endpoint, that final term is zero. In implementation terms this is still a Held-Karp-style bitmask DP for a Hamiltonian path, but the current implementation treats depot costs as first-class optional terms rather than as a future extension.

5. Cost model

The optimizer scores only transfer legs, using an energy-like fixed-wing cost

The DP itself does not draw geometry while it searches. Before the subset search starts, the backend evaluates connector candidates between every ordered pair of area traversal modes. When takeoff or landing endpoints are present, it also evaluates endpoint-to-area and area-to-endpoint connector costs. Those connectors include terrain-aware start and end altitudes, so the score captures more than 2D distance.

Transfer cost
C = thorizontal wh + tclimb wc + tdescent wd

Coverage quality inside each area is excluded here because it was already optimized in the earlier stages. The sequence optimizer changes transit order and flip choices; it does not re-plan strip spacing, camera triggering, overlap, GSD, or exact-region coverage.

6. Complexity

DP does not remove the combinatorics, but it compresses them dramatically

The brute-force space for 11 areas is 81,749,606,400 candidates. The DP collapses those into 22,528 valid states by merging route prefixes that reach the same visited-set / terminal-area / flip-state summary. For this formulation, the exact state count is N · 2N: each subset contributes one state for every possible terminal area inside that subset, and each of those terminal choices already carries the forward-versus-flipped mode.

In product terms, that growth is why the backend uses exact DP only up to a configured mission size. The current default and validation cap for exactSearchMaxAreas is 17 areas, which keeps the known 17-area Wingtra mission on the exact path by default. Larger missions, or requests that set a lower exact-search limit, use the greedy fallback to keep latency bounded.

8. Fallback path

Beyond 17 areas, the fallback stays depot-aware

Exact DP is the preferred path when the mission is within the configured limit, because it gives the true optimum under the current transfer-cost model. For larger missions, the backend uses a greedy fallback. The important current detail is that this fallback optimizes the same depot-aware objective rather than building a free-start path and scoring the depot legs only after the fact.

The fallback starts with the single area and flip state that minimizes the full one-area mission cost, including the optional takeoff and landing legs. It then inserts each remaining area at the cheapest position in the current sequence, evaluating every insertion against the full path delta. Insertions at the front use the takeoff edge when one exists; insertions at the back use the landing edge when one exists. After the initial construction, the backend still runs local flip and swap improvement passes, and those passes are scored with the same full objective.

The fallback is deliberately much stronger than a simple nearest-neighbor route builder, but it is still not exact. This distinction matters most on imported Wingtra missions with a home takeoff and landing point. A greedy sequence that ignores those anchors while it is being built can choose a path whose first and last areas are convenient for inter-area travel but expensive to reach from home. The current fallback avoids that class of mistake by making the depot edges part of every construction and refinement decision.

9. Conclusion

What this DP layer contributes to the overall mission pipeline

The key design choice is separation of concerns. One set of algorithms decides how to split the terrain and what bearing each area should use. This sequencing layer assumes that work has already been done and solves the remaining combinatorial problem: in what order should the fixed areas be flown, which ones should be reversed, and how should optional takeoff and landing anchors influence the first and last legs?

That is what makes this a good engineering use of dynamic programming. The state is compact, the overlap between subproblems is large, and the exact answer is feasible on realistic mission sizes up to the current threshold. When the mission is larger, the depot-aware greedy fallback uses the same cost model so the product still optimizes the right objective, even though it no longer has an exact optimality certificate.
